# ロッカー抽選 CLI 設計書

## 1. 目的

大学内ロッカーの週次申請処理を、管理者向けのクロスプラットフォーム CLI として実装する。開発は Python で行い、配布時は Python 環境不要の実行ファイルとして `Linux` / `Windows` / `macOS` で動作できる構成を目指す。

## 2. 前提

- 管理者 UI は不要とし、運用は CLI と CSV / PDF ファイルで完結させる。
- 週次処理は `validate` と `lottery` の 2 コマンドに分ける。
- 自動判定結果と人手確認用ファイルは分離する。
- 管理者は `review` 配下の CSV に `manual_status` を記入して抽選対象を確定する。
- 手動復帰は行わない。自動判定で `invalid` に入った応募はそのまま無効とする。

## 3. 入力データの想定

`demo-input/applicant_data.csv` と `demo-input/partner_data.csv` を現時点のサンプル入力として扱う。

### 3.1 申請者 CSV の主要列

- `タイムスタンプ`
- `申請者の学籍番号`
- `申請者の氏名`
- `申請者の学生証写真`
- `共同利用者の有無`
- `共同利用者の学籍番号`
- `共同利用者の氏名`
- `階数希望選択（共同利用者なし）`
- `階数希望選択（共同利用者あり）`

### 3.2 共同利用者 CSV の主要列

- `タイムスタンプ`
- `共同利用者の学籍番号`
- `共同利用者の氏名`
- `共同利用者の学生証写真`

### 3.3 補足

- サンプル CSV では学生証画像列に URL ではなく `accept` / `denied` のような値が入っている。実装では列の中身を文字列として保持し、表示・確認対象として扱う。
- 先頭に BOM が含まれる可能性があるため、CSV 読み込みは BOM 対応が必要。

## 4. 配布・実行方式

### 4.1 開発時

- Python による CLI アプリケーションとして実装する。
- 依存関係は最小限に抑える。
- 設定値は `JSON` か `YAML` の設定ファイルに外出しする。

### 4.2 配布時

- `PyInstaller` による単一実行ファイル化を第一候補とする。
- 実行ファイルは OS ごとにビルドする前提とする。
- GUI や OS 固有 API への依存は持たない。

## 5. コマンド設計

### 5.1 `validate`

役割:

- 週次入力 CSV を読み込む
- 応募データを正規化する
- E1〜E4 の自動判定を行う
- 自動判定の保存用 CSV と人手確認用 CSV を出力する

入力:

- 申請者 CSV
- 共同利用者 CSV
- 年度内の当選履歴 CSV
- ロッカー確定表 CSV
- 設定ファイル
- 対象期間 ID (`termX`)

出力:

- `output/termX/validation/valid_2F.csv` 〜 `valid_6F.csv`
- `output/termX/validation/invalid_app.csv`
- `output/termX/validation/invalid_par.csv`
- `output/termX/review/review_2F.csv` 〜 `review_6F.csv`

### 5.2 `lottery`

役割:

- 管理者確認済みの `review_*.csv` を読み込む
- `manual_status=keep` のみを抽選対象にする
- 空きロッカー数に応じて階別抽選を行う
- ロッカー番号を割り当てる
- 当選履歴、ロッカー確定表、PDF を出力する

入力:

- `output/termX/review/review_2F.csv` 〜 `review_6F.csv`
- 当選履歴 CSV
- ロッカー確定表 CSV
- 設定ファイル
- 対象期間 ID (`termX`)

出力:

- `output/termX/lottery/result.csv`
- `output/termX/lottery/locker_assignments.csv`
- `output/termX/lottery/lottery_result_YYYYMMDD.pdf`
- `output/termX/lottery/lottery_log.csv` または同等のログ

## 7. データモデル

### 7.1 正規化後の応募モデル

1 レコードは 1 申請を表す。主な保持項目:

- 申請タイムスタンプ
- 申請者氏名
- 申請者学籍番号
- 申請者学生証情報
- 利用人数
- 希望階
- 共同利用者氏名
- 共同利用者学籍番号
- 共同利用者学生証情報
- 共同利用者タイムスタンプ
- 自動判定結果

### 7.2 当選履歴

列:

- 申請者氏名
- 申請者学籍番号
- 共同利用者氏名
- 共同利用者学籍番号
- 処理日
- 割り当てロッカー

用途:

- E3 判定
- 年度内の再応募防止

### 7.3 ロッカー確定表

列:

- `ロッカー番号`
- `割り当て状態`

必要に応じて内部表現では追加情報を持てるが、外部出力は要件通り 2 列にする。

## 8. バリデーション仕様

### 8.1 E1: フォーマット / 必須欠落

以下を無効とする:

- 学籍番号が 7 桁数字でない
- 学籍番号が指定パターンに一致しない
- 必須項目が欠けている
- 希望階が取得できない
- 階と利用人数の組み合わせがルール違反

学籍番号の許容パターン:

- `15` + 数字 5 桁
- `4` または `8` + `1`〜`6` + 数字 5 桁

希望階は以下のどちらか非空の列から取得する:

- `階数希望選択（共同利用者なし）`
- `階数希望選択（共同利用者あり）`

### 8.2 E2: 共同利用者不在

- 2 人利用の応募で、申請者フォーム記載の共同利用者学籍番号が対象期間内の共同利用者 CSV に存在しない場合は無効
- 対象期間内に申請者・共同利用者の双方が揃っていなければ無効

### 8.3 E3: 当選履歴重複

- 年度内の既当選者が申請者または共同利用者に含まれる応募を無効とする

### 8.4 E4: 多重応募 / 競合ペア

- 同一人物の複数提出は、有効かつ最新の提出のみ採用
- 異なる複数ペアに同一人物が含まれる場合は、競合する有効データのうち最も早いペアを採用
- ペアの早さ判定は共同利用者タイムスタンプを申請者タイムスタンプより優先する

### 8.5 エラーコードの優先順

- `E1` → `E2` → `E3` → `E4`
- 複数該当時は最小番号のコードを採用する

## 9. 人手確認仕様

- `validate` 実行後、`review/review_2F.csv` 〜 `review_6F.csv` を出力する
- `review` CSV には `manual_status` 列を追加する
- 管理者は学生証情報を確認し、各行に `keep` または `reject` を記入する
- `lottery` は `manual_status=keep` のみ抽選対象にする
- 空欄や不正な値は安全側に倒し、抽選対象外として扱う

## 10. 抽選仕様

- 階ごとに独立して抽選する
- 空き枠数はロッカー確定表から算出する
- 応募数が空き枠以下なら全員当選
- 応募数が空き枠超過なら無作為抽選
- 1 人利用も 2 人利用も消費枠は 1
- 割り当てロッカーは階内の空き番号昇順で付与する
- 抽選の再現性確保のため、乱数シードを指定可能にする

## 11. PDF 出力仕様

- 1 ファイルに 2F〜6F をセクション分けして出力する
- タイトルは `ロッカー抽選結果（YYYY年MM月DD日）`
- 公開情報は申請者学籍番号のみ
- 当選者なしの階は `該当なし` と表示する
- フォントやレイアウトは 3 OS で崩れにくい単純な構成を優先する

## 12. エラーハンドリング

- 入力ファイル欠落時は明確なエラーメッセージを出して終了する
- 列名不足時は必要列一覧を表示して終了する
- `review` CSV の `manual_status` 不正値は警告を出し、その行を除外する
- 既存の出力先がある場合は上書き方針を明示する

## 13. テスト方針

- 学籍番号形式の単体テスト
- 階と利用人数制約の単体テスト
- 共同利用者照合の単体テスト
- 当選履歴チェックの単体テスト
- E4 競合解決の単体テスト
- `validate` コマンドの結合テスト
- `lottery` コマンドの結合テスト
- PDF 出力ファイル生成確認
- BOM 付き CSV の読み込み確認

## 14. 実装上の補足

- 設定ファイルに年度、階設定、ロッカー番号範囲、入出力ベースディレクトリ、乱数シード既定値を持たせる
- 実装は機能ごとに分割し、CLI 層とドメインロジックを分離する
- ユーザー配布向けには実行ファイル名と使用例を README に記載する
